# 自循环工作流: 源码验证 (Verification)

## 概述

给定文档中的声明与源码目录列表，Agent 以最小 token 代价直接比对源码，逐条验证报告中的每一项断言是否存在偏差。

> **核心哲学(源自 6 个真实 1:1 移植/验证工作流的教训)**: **假设端口/报告是错的**。实战数据: agent 端口 1:1 率约 60%，修复轮自身也会引入新 bug。验证 agent 必须是偏执的审计者，绝不盲信任何"已 1:1/已修复"的自报。

---

## 入口输入

```yaml
doc_path: "path/to/report.md"
src_dirs: ["path/to/src1", "path/to/src2"]
skip_patterns: ["*.meta", "*.asset"]  # 忽略的文件
output_path: "./verify_result.md"
round: 1                    # 验证轮次; round>=2 时启用修复轮专用检查
```

---

## 核心 prompt (最小化模板)

```
你收到一份技术报告和源码目录,请验证报告的每项断言。

**假设这个实现是错的**；带敌意找出对真源的每一处偏离并就地修复。
验证 agent 独立于实现 agent，绝不盲信任何"已1:1/已修复"自报。
—— 历史实战: agent 端口 1:1 率约 60%；修复轮自身也会引入新 bug / 过度修正 / 回退授权偏离 / 破坏语法。

验证规则:
1. 对报告中每行标注 `文件:行号` 的断言,打开对应文件验证
2. 对未标注行号的断言,在源码目录中 grep 搜索确认
3. 验证结果分三级:
   ✅ = 源码完全吻合（**逐字逐运算符核对，不看表面**）
   ⚠️ = 部分吻合(值有偏差/文件路径不同/行号偏移)
   ❌ = 源码中不存在或矛盾
4. 对 ❌ 和 ⚠️ 项,输出: 报告声明 | 源码事实 | 差异说明 | 运行期/画面症状 | **就地修复**

**真 bug 修复规则**:
  - 每处真偏离：伪码/端口 file:line + 真源 file:line + 症状 + 就地 Edit 修复
  - 修复时保留中文注释，标"源=文件:行"
  - 不删用户已扩展字段/不碰范围外文件
  - 确为 1:1 给逐步 file:line 对应表证明等价，不盖橡皮章

**从历史教训总结的 11 类必查偏离模式**(逐条打勾)：

1. **[反空心] 核函数空心化**: 复杂核是否被替换为简化式？骨架对但核心数学步骤缺失或简化 → 必须逐行核每个数学步骤都在
2. **[反假注释] 假注释**: 注释声称"已1:1 / 源默认X / 这里等价"但实际代码不符？→ 以真源为准，发现即改注释或删
3. **[反常量漂移] 常量/魔数/边界漂移**: clamp 区间是否和真源一致？clamp 上界/下界是否偏移？< vs <= 是否一致？默认值是否和真源一致？→ 历史 bug 案例：springPower clamp 上界 0.2f→1.0f(5×放宽)、backstopRadius 下界 0.0f→0.1f(打穿)、weight 默认值 1f→0f(默认开关态反)
4. **[反方向] 方向/法线/位移符号反转**: a-b 是否抄成 b-a？(毁灭性，历史高危 #1)
5. **[反配对] 写/读侧 key 不一致**: toroidal 寻址、cell-key 公式、通道编码在写侧和读侧是否逐字节一致？→ 不一致则 GI 永远采错 cell
6. **[反配对] 缓存 store/read 不配对**: store saturate(p/X) ↔ read ×X 值是否匹配？(如 PTGI 2.4 配对)
7. **[反配对] ping-pong 方向反**: 写 alt 读 main 是否搞反？
8. **[反假 gate] 假 gate / if-false 禁用**: `#if false / #if 0 / if(0)` 是否屏蔽了真代码段？→ 历史案例: 摩擦更新整段 `#if false ... // とりあえずOFF` 强制禁用
9. **[反臆造] 字段/API/常量臆造**: 是否为真源没有的字段/常量/方法？字段名是否和真 struct 每个字段一致？→ 必须从真源逐个核对，不凭"感觉应该有"
10. **[反缺失] 字段/枚举值/方法缺失**: 真源有的字段/枚举值/方法在端口是否遗漏？→ 序列化错位(历史案例 BUG-5: 字段偏移)
11. **[反缺失] 估计器积分域未闭合**: uniform→cosine 是否乘以 π/4？非对称采样权重是否补偿？

**修复轮专用检查(round>=2)**:
12. **[反过度] 过度修正**: 第一轮改的对的项是否被改回错？
13. **[反语法] 语法破坏**: 类型不匹配/缺分号/括号不配/swizzle 越界/未声明符号/残留 GLSL 特有语法？
14. **[反越界] 越界修改**: 是否修改了本文件/本领域范围外的文件？
15. **[反回退] 误删/改回授权偏离**: 已授权的同义替换是否被"修回去"？

文件列表: {src_dirs 展开为具体文件清单}

注意:
- 不只读 struct/enum/const 声明——**还必须读 method 体关键步骤**(反空心)
- 跳过 {skip_patterns} 匹配的文件
- 总读取行数控制在 2000 行以内
- 优先验证常量/枚举值/结构体字段布局
- 跨文件配对验证(写 vs 读、store vs read)必须两个文件同时看完才能下结论
```

## 验证清单格式

每项验证输出格式:

```markdown
### 断言 #{idx}: {章节名}

**报告声明**: "{报告原文}"  
**源码位置**: {文件}:{行号}  
**源码事实**: {源码中的实际内容}  
**判定**: ✅/⚠️/❌  
**差异**: {如有,说明具体差异}  
**偏离类别**: {反空心/反假注释/反常量漂移/反方向/反配对/反假 gate/反臆造/反缺失/修复轮语法/修复轮越界/修复轮过度}
```

## 批处理模式

当验证范围涵盖多份文档时:

```
阶段 1: 提取所有含 "文件:行号" 的断言 → 去重 → 按文件分组
阶段 2: 对每个文件,按行号排序 → 一次打开 → 用 offset 批量验证
阶段 3: 对无行号的断言,按关键词 grep (每个关键词只 grep 一次)
阶段 4: 跨文件配对验证——对每个"写/读配对"断言,同时读两个文件确认一致性
阶段 5: 汇总报告

约束:
- 同一文件只打开一次
- 连续的行号范围合并为单次 offset+limit 读取
- grep 结果缓存: 同一关键词不重复搜索
```

**示例**:
```
文件 group: GBufferOutput.cs
  断言 L7-23  struct 布局 → 一次读取 L1-30 完成 3 项验证
  断言 L26    GetGBufferAttachment(GBufferID) → 已在同次读取中
文件 group: BinningPass.cs
  断言 L9-24  BinningData → 一次读取确认 7 字段
```

## 汇总防截断规则

```
汇总阶段必须覆盖全部验证单元，严禁因 token 预算截断静默丢单元。

策略:
1. 对每个单元取紧凑投影：只保留 {cell, verdict, deviationsFound, fixesApplied, deviationCategory}
2. 丢弃巨大的 correspondence 全表（每单元截 ~1600 字上限）
3. 汇总 agent 的输入大小通过紧凑投影控制在 8000 字以内
4. 若仍超限 → 分批汇总: 先汇 5 组中间结果，再汇最终
```

---

## 容错处理

```
- 若文件不存在: 在 src_dirs 中按 basename 递归搜索
- 若行号不匹配(IFix 编译导致偏移): 取文件前 30 行确认类型声明区域
- 若 enum 值被 IFix 擦除: 从相邻 assembly 指令反推值
- 若结构体字段名不可读: 记录 "字段名不可读" 而非猜测
- 若真源方法体无 IL(IL2CPP dump): 只核对签名 + 外部调用序，标注"dump 无体、已核签名+调用序"
```

---

## 自循环退出条件

```
循环验证直到满足以下条件之一:
1. 所有断言均已验证(无 ❌ 剩余)
2. 剩余 ❌ 均标注 "源码中无法确认" 且经过 2 轮复查
3. 超出最大 token 预算
```
